# Box 看板重构设计

**日期：** 2026 年 7 月 17 日
**状态：** 已确认，待实施

## 目标

将当前按固定业务栏目展示的首页重构为可持久化的个人 Box 看板。用户可以创建、重命名、拖动、缩放和删除 Box；每个任务只归属一个 Box。任务在 Box 内以简洁的待办清单呈现，支持快速新建、勾选完成、行内重命名和抽屉式详情编辑。

本次改造保留 React + FastAPI + 本地 JSON 存储架构，并安全迁移已有私有数据。

## 已确认的产品决策

- 一个任务只能属于一个 Box。
- Box 使用对齐网格：拖动时吸附到网格；缩放按网格列、行变化。
- 初始全新看板只创建一个 `Inbox`；`Inbox` 可以和其他 Box 一样重命名、拖动、缩放、删除。
- 删除 Box 时用户二选一：删除其任务，或迁移全部任务至一个现有 Box。
- 任务有一个可选的截止日期，仅选择日期，不选择具体时刻或日期范围。
- 任务详情仍在右侧抽屉编辑，不跳转独立页面。
- 标签为多值字段。
- checkbox 只负责完成/恢复；点击任务标题行内重命名；点击右侧铅笔图标打开任务抽屉。
- 旧数据升级后全部迁入自动创建的 `Inbox`。

## 信息架构与数据模型

### Box

```ts
interface Box {
  id: string
  title: string
  layout: {
    x: number
    y: number
    w: number
    h: number
  }
  createdAt: string
  updatedAt: string
}
```

- `x/y/w/h` 是桌面端网格布局的持久化位置与尺寸。
- 标题在 Box 顶部可编辑。
- Box 没有系统保留属性，任何 Box（包括 Inbox）都可删除。

### Task

```ts
interface Task {
  id: string
  title: string
  boxId: string
  tags: string[]
  priority: 'high' | 'medium' | 'low' | null
  dueDate: string | null // YYYY-MM-DD
  details: string | null
  completedAt: string | null
  position: number
  createdAt: string
  updatedAt: string
}
```

- `completedAt === null` 表示未完成；有值表示已完成。系统不再存储独立 `status`。
- `position` 用于在同一完成分组内维持稳定顺序；呈现时未完成任务永远在前，已完成任务永远在后。
- UI 不再暴露旧字段 `type`、`area`、`nextAction`、`notes`、`horizon`、`status`。
- 旧的 `type` 与 `area` 迁移后成为标签；旧的 `nextAction` 与 `notes` 合并为 `details`。

### Board document

存储层使用有版本号的看板文档：

```json
{
  "version": 2,
  "boxes": [],
  "tasks": []
}
```

现有任务列表属于旧格式。首次加载新格式时，迁移器会识别旧格式并建立新文档。

## 主界面

### Box 网格

- 主视图为桌面优先的对齐网格，而不是旧的固定分区。
- 用户拖动 Box 时显示网格吸附效果；拖动结束后立即保存布局。
- 用户通过 Box 边缘/角落调整尺寸；尺寸变化后立即保存。
- 页面提供“新建 Box”入口。新 Box 使用用户输入的名称，并放在第一个可用网格位置。
- 小屏设备降级为单列 Box 列表：保留内容、快速新增和编辑；不提供容易误触的自由拖拽/缩放。

### Box 头部

- 显示可行内编辑的标题。
- 右上角提供新增按钮。
- 新增按钮在 Box 内容区展开一个紧凑输入框；输入标题并按 Enter 即创建任务；Esc 或失焦取消空输入。
- Box 操作菜单包含重命名和删除。删除时先显示任务处理选择，再执行删除或迁移。

### 任务列表

任务以紧凑任务清单显示：

```text
☐ 完成首页重构                              ✎
☑ 已确定信息架构
```

- 点击 checkbox：切换完成状态；完成任务灰色、删除线，并排在该 Box 的未完成任务之后。
- 点击任务标题：进入行内重命名；Enter 保存，Esc 取消；失焦提交非空标题。
- 点击铅笔：打开右侧任务抽屉。
- checkbox、标题编辑、铅笔分别阻止事件冒泡，避免互相触发。
- 不以“完成时间”重新排序同一分组，避免频繁操作让用户失去任务位置感。

## 右侧任务抽屉

抽屉继续作为完整任务编辑界面，字段按以下顺序呈现：

1. 标题；
2. 标签（可输入、创建、移除多个标签）；
3. 优先级（高 / 中 / 低）；
4. 截止日期（原生日历选择器，仅日期）；
5. 所属 Box（下拉框，可移动任务）；
6. 详情（多行文本，合并“下一步动作”和“备注”）；
7. 删除任务。

抽屉不再显示状态、领域、类型、下一步动作和备注。完成状态在任务行 checkbox 上管理；抽屉可以显示但不重复提供第二套状态控件。

## API 与后端边界

### Box API

- `GET /boxes`：列出所有 Box 与布局。
- `POST /boxes`：创建 Box。
- `PATCH /boxes/{id}`：重命名和/或保存布局。
- `DELETE /boxes/{id}`：请求体包含 `taskDisposition`：`delete` 或 `move`；后者必须包含有效的 `targetBoxId`。

### Task API

任务 API 围绕新模型读写：

- 列表、创建、更新、删除；
- 更新可原子修改完成状态、标题、Box 归属、标签、优先级、截止日期和详情；
- 创建任务必须指定 `boxId`；
- 后端验证 `boxId` 及 Box 删除时的 `targetBoxId` 存在性。

对前端的接口命名将统一使用 Task；如现有 `/threads` 端点有兼容性需求，可在迁移期保留仅服务于旧客户端的适配层，而新前端不得继续依赖旧字段。

## 迁移与数据安全

1. 启动时检查目标看板文档；如果不存在且检测到旧任务 JSON，开始一次性迁移。
2. 创建名为 `Inbox` 的 Box，给它稳定的默认网格位置与尺寸。
3. 每条旧记录成为一个 Task，并统一写入该 Inbox。
4. 字段映射：
   - `title` 保留；
   - `type`、`area` 的非空值去重后写入 `tags`；
   - `priority`: `now → high`、`next → medium`、`later → low`；
   - `nextAction`、`notes` 合并为 `details`，两者都有内容时以清晰的段落标题区分；
   - `status === done` 变为已完成，其余状态变为未完成；
   - 旧日期相关字段不承载精确截止日期，因此 `dueDate` 留空；
   - 可用的创建、更新时间予以保留。
5. 写入前创建旧 JSON 备份；新文档通过临时文件加原子替换写入。
6. 数据迁移必须幂等：成功迁移后的后续启动不重复创建 Inbox 或重复导入任务。

## 视觉与交互原则

- 采用轻量卡片、足够留白和清楚层级，而非旧首页的密集固定分栏。
- 信息层级：Box 标题与新增操作优先，任务标题次之，标签/日期/优先级仅在需要时轻量显示。
- 完成状态具有足够明显的灰度与删除线，但不降低勾选恢复的可读性。
- 加载、空 Box、保存失败、删除确认和迁移失败必须有清楚反馈。

## 验收与测试

后端：

- 旧数据迁移正确且幂等；
- Box CRUD、布局校验与删除迁移/删除策略；
- Task 的 `boxId` 验证、完成状态、字段校验与日期格式。

前端：

- 任务未完成/已完成的排序；
- checkbox 完成切换；
- 快速新增与行内重命名；
- 抽屉字段与多标签/日期选择；
- Box 删除策略、布局保存和关键空/错误状态。

端到端手工验证：

- 从已有 JSON 自动生成 Inbox 并保留原任务；
- 新建、拖动、缩放、重命名和删除 Box；
- 新建任务、完成置底、重命名、编辑和移动任务；
- 刷新页面后布局与数据保持一致。

## 非目标

- 不引入多用户、登录、云同步或数据库；
- 不实现任务属于多个 Box；
- 不实现带具体时间的日程、开始/结束时间段或周期性任务；
- 不在移动端实现桌面级拖拽缩放。
