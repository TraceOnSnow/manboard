## Context

现有 dashboard 使用单页、九个固定分区、同一种卡片和阻断式编辑弹窗。它已能完整读写 Thread，但不适合两个高频时刻：突然冒出想法时的捕捉，以及每天打开时决定先做什么。此次变更只改前端体验；核心模型、FastAPI API 与 JSON storage 已足以承载第一版。

用户确认的产品性格是“行动快速、简约、明快、chill、cool”。其真实行为不是严格排程，而是随手记录、勾掉小事、在自然浏览中重新发现久未触碰的项目或阅读内容。

## Goals / Non-Goals

**Goals:**

- 让标题级的想法能在一次输入后进入 Inbox。
- 让首页首先回答“今天有什么小事可以做”，同时保留全部个人内容的低压力浏览入口。
- 为即时待办和长期内容提供不同复杂度的状态交互。
- 用一致、响应式且可访问的视觉系统替换粗糙的卡片堆叠。
- 交付可实际使用的第一版，以真实反馈决定后续的统计、提醒和信息架构。

**Non-Goals:**

- 不修改 Thread 后端模型、API、storage 或新增数据库。
- 不实现浏览器通知、自动提醒、搜索、图表分析、认证、路由型多页面或外部同步。
- 不预先设计空洞的统计页；侧边导航先作为分区定位与未来扩展的应用外壳。
- 不支持自定义分区、拖拽排序或批量操作。

## Decisions

### D1: 用既有 `type=other` 表示 Inbox

全局快速记录创建 `type=other`、`horizon=none` 的 Thread；前端将这组条目筛选为 Inbox。用户在详情抽屉中通过修改既有字段完成“整理”。

这避免为仍在探索中的体验新增后端字段或迁移。替代方案是新增 `inbox` type 或 `capturedAt` 字段；两者会扩大模型与 API 范围，但在尚未验证捕捉流程前没有必要。

### D2: 首页以行动区为首，内容区作为可下钻的连续浏览

顶部固定为快速记录，其后依次为今日待办和 Inbox。长期内容维持 type/horizon 驱动的分区，项目、研究和媒体以更新时间倒序排列。桌面端侧边导航定位主要板块；移动端移除常驻导航，保留单列内容和明显的快速记录。

这比把九个空分区全部平铺更符合“每天打开”和“闲时往下翻”的行为，同时不会隐藏任何既有数据。

### D3: 按条目意图决定交互密度

今日待办只显示复选框，勾选直接调用现有 PATCH API 将状态在 active/done 间切换。长期卡片显示状态菜单而非固定顺序循环，并在被点击时打开右侧详情抽屉；抽屉复用表单字段、负责编辑和删除确认。

替代方案是所有条目共用四态标签或用拖拽移动状态。前者增加维护成本且循环不透明，后者对移动端和键盘用户不友好。

### D4: 视觉系统采用“中性色 + 行动蓝”的扁平工具风

设计 token 使用近白背景、深墨色文字、蓝色主要行动色和语义化状态色；卡片使用轻边框、统一圆角和极弱或无阴影。字体使用 Plus Jakarta Sans，图标使用一个一致的 SVG 图标库。交互动效仅用于 150–200ms 的 opacity/transform 反馈，并实现 `prefers-reduced-motion`。

该方案取 UIPro 的高可读性、标准信息密度、无障碍焦点与低动效建议，但刻意舍弃其偏落地页的大 Hero 和营销 CTA 模式。

### D5: 侧边栏是导航与扩展边界，不是伪造功能页

首版桌面外壳有精简侧边导航，用于定位 Today、Inbox、Projects 和其他内容区；统计与更多页面不预先实现。条目详情使用右侧抽屉，而不是把详情塞进导航。

这样既给未来页面留出自然入口，又不以空统计或多路由拖慢第一版。

## Risks / Trade-offs

- **[Inbox 复用 other 语义过宽]** → Inbox 仅筛选 `other + none`，并在用户有实际整理需求后再评估增加专用字段。
- **[今日待办和普通待办边界不清]** → 创建入口会预设 today；详情中仍可修改 horizon，后续以真实使用反馈决定是否需要专用 quick-task type。
- **[侧边导航增加桌面复杂度]** → 保持为少量锚点，移动端不常驻，且不实现未经验证的功能页。
- **[引入网页字体影响首次加载]** → 使用 `font-display: swap` 并保留系统字体回退。

## Migration Plan

1. 增加设计 token、图标依赖与应用外壳，不改变 API。
2. 将现有 `App` 拆分为快速记录、分区、条目卡片和详情抽屉组件。
3. 基于现有字段实现 Inbox、今日待办和长期条目排序；已有数据无需迁移。
4. 在样例数据和真实私有数据路径下验证创建、整理、状态更新、删除与刷新持久化。
5. 若体验不适合，可回滚前端提交；后端与数据文件保持兼容。

## Open Questions

- 统计页、提醒和搜索是否值得做，必须在第一版被真实使用后再决定。
- Inbox 是否最终需要独立于 `other` 的数据语义，留待观察用户是否频繁整理或希望为其他用途保留 `other`。
